# teamAgent — 基于 AutoGen 的自动化项目开发团队

## 项目概述

**项目名称**: teamAgent

**一句话描述**: 基于 AutoGen 0.4+ 构建的多智能体系统，通过三个 AI 角色（开发经理、开发人员、测试人员）的自动化协作，将用户的任意开发任务从需求转化为提交到 GitHub 的完成代码。

**核心目标**:
1. 用户只需输入一个开发任务的自然语言描述，系统即可自动完成从需求分析到代码提交的全流程
2. 三个 Agent 角色模拟真实软件开发团队的协作流程，通过 GitHub Issue 作为通信媒介
3. 作为一个通用框架，能够处理任意类型的软件开发任务
4. 用户不深度介入，但保留关键节点的确认机制

## 技术栈

| 项目 | 规格 |
|---|---|
| 语言 | Python 3.10+ (3.11.6) |
| 异步运行时 | asyncio (AutoGen 0.4+ 原生异步) |
| `autogen-agentchat` | >=0.5.7 — 高层 Agent API |
| `autogen-core` | >=0.5.7 — 事件驱动消息层 |
| `autogen-ext[openai]` | >=0.5.7 — OpenAI 兼容客户端 (连 DeepSeek) |
| `PyGithub` | >=2.x — GitHub REST API |
| `gitpython` | >=3.x — 本地 Git 操作 |
| **模型** | DeepSeek API (`deepseek-chat`) |
| **连接方式** | `OpenAIChatCompletionClient` + OpenAI 兼容接口 |

> ⚠️ 注意事项:
> - `deepseek-reasoner` 不支持 tool calling，优先使用 `deepseek-chat`
> - AutoGen 已于 2025 年末并入 Microsoft Agent Framework (MAF)。如面向生产环境建议评估 MAF；如用于快速原型，autogen-agentchat 0.5.x 仍是成熟稳定选择

## 角色详细定义

### 1. 开发经理 (Manager)

**职责**: **需求澄清确认（阻塞关卡）** → 需求分析 → 任务拆解 → 创建 Issue → 分配任务 → 审查 PR → 合并代码 → **迭代评估 → 下发新一轮 Issues（循环）**

**系统提示词核心要求**:
- 你是项目的开发经理，负责统筹整个开发流程
- **⚠️ 首要职责：需求澄清**。收到用户需求后，你必须先在父 Issue 中与用户逐条确认所有模糊点。在所有需求细节明确、用户回复"确认"之前，你**绝对不能**创建子 Issue 或进入开发流程
- 你需要将用户输入的模糊需求拆解为清晰的任务列表和验收标准
- 每次 PR 合并后，你必须执行迭代评估：回顾已完成工作、对照原始需求、评估剩余工作
- **如果需求未完全实现，你必须创建新一轮的子 Issues，推动开发循环继续**
- **只有当所有验收标准全部满足时，你才能关闭父 Issue 并宣告任务完成**
- 当你判断项目需求彻底完成时，输出总结报告并发送"任务完成"信号
- 你需要为每个任务规划技术方案、涉及的文件、接口设计
- 你需要将任务以 Issue 的形式创建到 GitHub 仓库
- 你负责分配任务给开发人员，并设置合理的优先级和截止时间
- 你需要在开发完成后审查 Pull Request，确认是否符合需求规格
- 你拥有最终合并 PR 的决策权
- 当开发或测试环节遇到阻塞时，你需要做出决策（回退、修改方案、或继续）
- **你需要在迭代循环中保持全局视角，确保每一步都在朝最终目标前进，而非陷入局部修改循环**

**拥有的工具/能力**:

| 工具名称 | 功能 |
|---|---|
| `create_issue` | 在 GitHub 仓库创建 Issue，包含标题、描述、标签 |
| `comment_on_issue` | 在指定 Issue 上添加评论 |
| `close_issue` | 关闭已完成或废弃的 Issue |
| `review_pull_request` | 审查 PR 代码，提交 Review 评论 |
| `merge_pull_request` | 合并通过的 PR 到 main 分支 |
| `create_branch` | 基于 main 分支创建功能分支 |

**输入**: 用户原始任务描述
**输出**: GitHub Issues（含需求规格、技术方案、验收标准）、PR Review、Merge 决策

### 2. 开发人员 (Developer)

**职责**: 领取 Issue → 克隆仓库 → 创建分支 → 编码实现 → 提交推送 → 创建 PR

**系统提示词核心要求**:
- 你是一名资深软件开发工程师，负责实际的编码实现
- 你会接收到开发经理分配的 Issue，其中包含详细的需求规格和技术方案
- 你需要在本地克隆仓库，基于功能分支进行开发
- 你遵循"小步提交"原则，每次完成一个独立的功能点就进行本地提交
- 你需要在代码中包含必要的注释和文档字符串
- 开发完成后，你需要创建 Pull Request 到 main 分支，并在 PR 描述中关联对应的 Issue
- 你负责根据代码审查意见和测试报告进行修改

**拥有的工具/能力**:

| 工具名称 | 功能 |
|---|---|
| `clone_repository` | 克隆远程仓库到本地工作目录 |
| `create_branch` | 基于 main 创建功能分支 |
| `read_file` | 读取项目中的文件内容 |
| `write_file` | 创建或修改文件 |
| `delete_file` | 删除文件 |
| `list_files` | 列出目录中的文件 |
| `git_commit` | 暂存更改并提交到本地仓库 |
| `git_push` | 推送分支到远程仓库 |
| `create_pull_request` | 创建 Pull Request，关联 Issue |
| `comment_on_issue` | 在 Issue 上汇报进度、提问 |
| `run_command` | 执行 Shell 命令（如运行测试、语法检查） |

**输入**: 被分配的 Issue
**输出**: 代码变更、Pull Request、进度汇报

### 3. 测试人员 (Tester)

**职责**: 获取 PR → 代码审查 → 功能验证 → 运行测试 → 提交 Review → 报告结果

**系统提示词核心要求**:
- 你是一名专业的 QA 测试工程师，负责确保代码质量
- 你会在开发人员创建 PR 后，第一时间拉取代码分支进行测试
- 你需要检查代码是否符合需求规格（参考对应的 Issue 描述）
- 你需要检查代码风格、命名规范、错误处理等质量维度
- 如果发现问题，你需要在 PR 上提交 Review Comment，并在关联 Issue 中描述 Bug
- 如果测试通过，你需要在 PR 上 Approve，并在 Issue 中报告测试通过
- 你的测试报告需要包含: 测试范围、测试方法、发现问题（如有）、测试结论

**拥有的工具/能力**:

| 工具名称 | 功能 |
|---|---|
| `fetch_pr_branch` | 获取 PR 对应的代码分支到本地 |
| `read_file` | 审查代码文件内容 |
| `run_command` | 运行测试命令、语法检查、lint 检查 |
| `submit_pr_review` | 提交 PR Review（COMMENT/APPROVE/CHANGES_REQUESTED） |
| `comment_on_issue` | 在 Issue 上报告测试结果 |
| `create_issue` | 如果发现 Bug，创建新的 Bug Issue 并关联 |

**输入**: 新建的 PR
**输出**: PR Review（Approve/Changes Requested）、测试报告、Bug Issue

### 核心循环机制

**开发经理的迭代评估职责**（这是整个循环的核心）:

每次成功合并一个/一批 PR 后，开发经理必须执行以下评估:

1. **回顾已完成的工作**: 本次迭代完成了哪些功能？关闭了哪些 Issue？
2. **对照原始需求**: 用户的总体需求中还有哪些部分没有实现？
3. **发现遗留问题**: 测试人员是否报告了新的 Bug？是否有技术债务需要处理？
4. **决定下一步行动**:
   - 如果需求未完全实现 → 创建新的子 Issues，开启下一轮开发循环
   - 如果发现新问题 → 创建 Bug Issues，进入修复循环
   - 如果所有需求已完成且无遗留问题 → 进入项目收尾

### 阶段详解

**阶段 0 — 需求澄清与确认（仅执行一次，阻塞后续所有流程）**:

> ⚠️ 此阶段是项目启动的**必须前置步骤**。只有在此阶段用户明确确认需求无误后，系统才能进入开发循环。

1. 用户输入项目总体需求描述
2. 开发经理分析需求，识别模糊点、歧义点、缺失信息
3. 开发经理在 GitHub 创建父 Issue，包含:
   - 对用户需求的初步理解和复述
   - **提出的确认问题清单**（标记为 `needs-clarification` 标签）
   - 建议的技术方向和架构思路
4. **用户在 Issue 评论区逐条回复确认问题**
5. **开发经理根据用户回复更新需求理解，如有新疑问继续提问**
6. **循环步骤 4-5，直到以下条件全部满足**:
   - 开发经理确认所有需求细节已清晰、无歧义
   - 用户明确回复 "需求确认完毕" 或 "可以开始开发"
7. 开发经理将父 Issue 标签改为 `confirmed`，移除 `needs-clarification`
8. 开发经理在父 Issue 中更新最终版需求规格 + 验收标准
9. 开发经理制定首轮迭代计划，拆解出第一批子 Issues
10. 子 Issues Assign 给开发人员，进入开发循环

**需求确认期间的规则**:
- ❌ 开发经理**不得**创建子 Issues 或 Assign 开发任务
- ❌ 开发人员/测试人员**不参与**此阶段
- ✅ 开发经理可以引用类似项目的经验给出建议
- ✅ 用户随时可以修改或补充需求
- ⏱️ 如果用户超过 48 小时未回复，开发经理可发送一次提醒；超过 7 天则暂停项目等待用户回归

**阶段 1 — 需求分析与任务规划（每次迭代执行）**:
1. 开发经理审视当前项目状态（已完成的 Issues、未关闭的 Issues）
2. 评估剩余工作量和优先级
3. 创建本轮迭代的子 Issues，标注优先级 (`priority/high` / `priority/low`)
4. 每个子 Issue 包含: 需求描述、技术方案、涉及文件、验收标准
5. Assign 子 Issues 给开发人员

**阶段 2 — 开发实现**:
1. 开发人员查看被 Assign 的 Issue
2. 克隆仓库 → 创建功能分支 (feature/{issue-number}-{short-desc})
3. 在 Issue 评论中汇报: "开始开发任务 #{number}"
4. 循环实现功能点（读文件 → 写文件 → 暂存 → 本地提交）
5. 遇到阻塞时在 Issue 中 @开发经理请求决策
6. 推送分支 → 创建 PR（描述中 Closes #{issue-number}）

**阶段 3 — 测试与审查**:
1. 测试人员查看新 PR → 拉取分支到本地
2. 执行代码审查 + 功能验证 + 需求对齐检查
3. 发现问题 → 提交 CHANGES_REQUESTED → 开发人员修复 → 重新审查
4. 测试通过 → 提交 APPROVE → 在 Issue 中报告测试结论

**阶段 4 — 合并与迭代评估（循环的决策点）**:
1. 开发经理确认 PR 通过测试审查
2. Merge PR → 关闭对应的子 Issue
3. **执行迭代评估**（见上方「核心循环机制」）:
   - **需要继续** → 回到**阶段 1**，下发新一轮 Issues
   - **项目完成** → 进入**阶段 5**

**阶段 5 — 项目收尾（仅执行一次）**:
1. 开发经理关闭父 Issue
2. 在父 Issue 最后一条评论中输出总结报告:
   - 完成的功能清单
   - 所有涉及的文件列表
   - 测试覆盖摘要
   - 已知问题/后续建议（如有）
3. 系统输出"项目需求全部完成"，终止运行

### 循环终止条件

系统在以下任一条件满足时退出循环:

| 条件 | 说明 |
|---|---|
| **需求全部完成** | 开发经理评估所有验收标准已满足，无遗留任务 |
| **最大迭代次数** | 迭代轮次达到上限（如 20 轮），防止无限循环 |
| **用户主动终止** | 用户通过 Issue 评论发送 "STOP" 指令 |
| **无法继续** | 开发经理判断当前技术条件无法完成剩余需求（如架构需要重构） |

### 异常处理

| 场景 | 处理方式 |
|---|---|
| 需求不明确 | 开发经理在 Issue 中提问，超时后使用默认假设 |
| 技术方案不可行 | 开发人员提出，开发经理评估后修改方案或回退 |
| 严重 Bug | 标记 blocking，开发人员优先修复，修复后重新进入开发循环 |
| 多次 Review 被拒 (>3次) | 开发经理介入决策：修改方案 / 放弃该 PR / 重新拆解任务 |
| Git 冲突 | 开发人员解决冲突后重新推送 |
| 无限循环 | 迭代轮次上限（默认 20 轮）+ `MaxMessageTermination(max_messages=50)` 双重保护 |
| API 失败 | 指数退避重试机制 |
| 迭代评估不准确 | 每 5 轮迭代强制输出一次需求完成度报告，方便用户审查 |

## GitHub 集成设计

### Issue 标签体系

| 标签 | 用途 |
|---|---|
| `type/task` | 开发任务 |
| `type/bug` | Bug 报告 |
| `type/question` | 需求确认问题（阶段 0 专用） |
| `status/needs-clarification` | 🔍 需求待澄清（阻塞开发流程） |
| `status/confirmed` | ✅ 需求已确认，可以开发 |
| `status/ready` | 待分配 |
| `status/blocked` | 阻塞中 |
| `status/done` | 已完成 |
| `priority/high` | 高优先级 |
| `priority/low` | 低优先级 |
| `agent/manager` | 由开发经理处理 |
| `agent/developer` | 由开发人员处理 |
| `agent/tester` | 由测试人员处理 |

### Issue 状态流转

**父 Issue（需求跟踪）**:
```
Open (needs-clarification) → 需求确认循环 → Confirmed → 子任务进行中 → 所有子任务 Closed → 父 Issue Closed
```

**子 Issue（开发任务）**:
```
Open → Assigned → In Progress → Review/Test → Closed
```

### PR 审查和合并流程

1. 开发人员创建 PR，设置 reviewers
2. 测试人员拉取分支 → 审查 → APPROVE 或 CHANGES_REQUESTED
3. 开发经理最终审查 → Merge (推荐 Squash merge)

### Agent 交互方式

- 通过 PyGithub 库操作 GitHub，工具函数封装为 AutoGen `@tool`
- Agent 通过 Team 聊天消息感知状态变化
- 评论内容通过 Tool 读取后作为消息注入到对话流

## 端到端测试场景

### 场景 1: 简单任务 (Smoke Test)
```
输入: "在仓库根目录创建一个 CONTRIBUTING.md 文件，包含基本的贡献指南"
预期:
1. 开发经理创建 1 个 Issue
2. 开发人员创建分支 → 写文件 → 创建 PR
3. 测试人员审查 → Approve
4. 开发经理合并 PR
5. 文件成功出现在 main 分支
```

### 场景 2: 中等复杂度任务 (功能开发)
```
输入: "在项目中添加一个 utils/math_helpers.py 模块，包含 add(a, b) 和 subtract(a, b) 函数，使用 pytest 编写测试"
预期:
1. 开发经理分解为子任务
2. 开发人员实现代码 + 测试
3. 测试人员运行 pytest → 确认通过 → Approve
4. 合并后仓库包含完整模块
```

### 场景 3: Bug 修复 (回归验证)
```
输入: "修复 main.py 中 print_hi 函数的拼写错误"
预期:
1. 开发经理创建 Bug Issue
2. 开发人员修复代码
3. 测试人员验证修复 + 回归检查 → Approve
4. 开发经理合并
```
