## 1. 简介

**teamAgent** 是一个基于 AutoGen 0.7+ 框架构建的多智能体系统。它模拟一个完整的软件开发团队，通过三个 AI 角色（开发经理、开发人员、测试人员）的自动化协作，将用户的自然语言开发需求转化为提交到 GitHub 的完成代码。

### 核心特性

- 🤖 **三角色协作**: 开发经理负责需求拆解和迭代管理，开发人员负责编码实现，测试人员负责质量保障
- 🔄 **迭代循环**: 自动评估项目进度，循环下发任务直到需求彻底完成
- 🔍 **需求澄清**: 开发前通过 GitHub Issue 与用户确认所有模糊需求
- 🛡️ **健壮性**: 内置重试、回退、冲突检测、超时控制
- 📊 **审计追踪**: 完整记录所有 Agent 行为和工具调用
- ⚡ **多任务并发**: 支持同时运行多个项目
- 🎨 **多后端**: 支持 DeepSeek / OpenAI / Ollama 多种模型

### 三个角色

| 角色 | 职责 | 拥有工具 |
|---|---|---|
| 🏗️ 开发经理 | 需求澄清 → 任务拆解 → 创建 Issue → 审查 PR → 迭代评估 → 合并 | 10 个 |
| 💻 开发人员 | 领取 Issue → 克隆仓库 → 编码实现 → 提交推送 → 创建 PR | 16 个 |
| 🧪 测试人员 | 拉取 PR → 代码审查 → 功能验证 → 提交 Review → 报告结果 | 14 个 |

## 2. 快速开始

### 2.1 环境要求

- Python 3.10+
- Git
- GitHub 账号 + Personal Access Token

### 2.2 安装

```bash
# 克隆项目
git clone <your-repo-url>
cd teamAgent

# 创建虚拟环境
python -m venv .venv

# 激活虚拟环境
# Windows:
.venv\Scripts\activate
# Linux/macOS:
source .venv/bin/activate

# 安装依赖
pip install -r requirements.txt
```

### 2.3 配置

编辑项目根目录的 `.env` 文件，填入你的配置：

```bash
# DeepSeek API（必填）
DEEPSEEK_API_KEY=sk-your-api-key
DEEPSEEK_BASE_URL=https://api.deepseek.com/anthropic
DEEPSEEK_MODEL=deepseek-chat

# GitHub（必填）
GITHUB_TOKEN=ghp_your-personal-access-token
GITHUB_REPO_OWNER=your-github-username
GITHUB_REPO_NAME=your-target-repo

# 可选配置
LOG_LEVEL=INFO           # 日志级别: DEBUG/INFO/WARNING/ERROR
MAX_ITERATIONS=20        # 最大迭代轮次
MAX_MESSAGES_PER_ROUND=50 # 单轮最大消息数
```

### 2.4 运行第一个任务

```bash
python main.py "创建一个 CONTRIBUTING.md 文件，包含基本的贡献指南"
```

系统会自动：
1. 在目标 GitHub 仓库创建需求确认 Issue
2. 等待你确认需求（在 Issue 中回复）
3. 确认后自动进入开发循环，直到任务完成
4. 生成执行报告到 `reports/` 目录

## 3. 环境配置

### 3.1 配置项说明

所有配置项在 `.env` 文件中设置：

| 变量 | 必填 | 默认值 | 说明 |
|---|---|---|---|
| `DEEPSEEK_API_KEY` | ✅ | — | DeepSeek API 密钥 |
| `DEEPSEEK_BASE_URL` | — | `https://api.deepseek.com/anthropic` | API 基础地址 |
| `DEEPSEEK_MODEL` | — | `deepseek-chat` | 模型名称（需支持 function calling） |
| `GITHUB_TOKEN` | ✅ | — | GitHub Personal Access Token |
| `GITHUB_REPO_OWNER` | ✅ | — | GitHub 用户名或组织名 |
| `GITHUB_REPO_NAME` | ✅ | — | 目标仓库名称 |
| `LOG_LEVEL` | — | `INFO` | 日志级别 |
| `MAX_ITERATIONS` | — | `20` | 最大迭代轮次（防无限循环） |
| `MAX_MESSAGES_PER_ROUND` | — | `50` | 每轮对话最大消息数 |

### 3.2 GitHub Token 权限

创建 Token 时需要以下权限：

- `repo` — 完整的仓库读写权限（创建分支、提交文件、创建 PR）
- `issues` — Issue 读写权限
- `pull_requests` — PR 读写权限

### 3.3 目标仓库

目标仓库**必须已经存在**于 GitHub 上。teamAgent **不会自动创建仓库**。

## 4. 命令行用法

### 基本用法

```bash
# 直接输入任务描述
python main.py "你的开发任务描述"

# 从文件读取任务
python main.py --file 任务描述.txt

# 进入交互模式
python main.py --interactive
```

### 完整参数

| 参数 | 说明 | 示例 |
|---|---|---|
| `<任务描述>` | 直接输入的任务文本 | `python main.py "创建 Web API"` |
| `--file <路径>` | 从文件读取任务 | `python main.py --file task.txt` |
| `--batch <文件...>` | 批量并发执行 | `python main.py --batch t1.txt t2.txt` |
| `--interactive` / `-i` | 交互模式 | `python main.py --interactive` |
| `--backend <后端>` | 切换模型后端 | `python main.py --backend ollama "任务"` |
| `--config <路径>` | 自定义角色配置 | `python main.py --config my.yaml "任务"` |

### 任务描述编写建议

任务描述应当清晰具体。示例：

```markdown
好的任务描述 ✅：

创建一个基于 FastAPI 的 Web 服务，包含以下功能：
1. 用户注册和登录接口（使用 JWT 认证）
2. CRUD 接口管理 Todo 列表
3. 使用 SQLite 作为数据库
4. 编写 pytest 测试用例
5. 添加 README 说明文档

差的任务描述 ❌：

做一个 Todo 应用
```

## 5. 工作流程说明

teamAgent 的运行流程分为 6 个阶段：

```
用户输入需求
    ↓
┌─────────────────────────────┐
│  阶段 0: 需求澄清 (阻塞关卡)  │ ← 开发经理通过 Issue 与你确认需求
│  ⛔ 确认前不进入开发流程      │
└─────────────────────────────┘
    ↓ 确认通过
┌─────────────────────────────┐
│  阶段 1: 任务规划            │ ← 开发经理拆解子任务、创建子 Issues
└─────────────────────────────┘
    ↓
┌─────────────────────────────┐
│  阶段 2: 开发实现            │ ← 开发人员编码、提交、创建 PR
└─────────────────────────────┘
    ↓
┌─────────────────────────────┐
│  阶段 3: 测试审查            │ ← 测试人员审查代码、运行测试
└─────────────────────────────┘
    ↓
┌─────────────────────────────┐
│  阶段 4: 合并与迭代评估      │ ← 开发经理合并 PR，评估进度
│                              │
│  ├─ 还有剩余工作 → 回到阶段 1 │
│  └─ 全部完成 → 进入阶段 5    │
└─────────────────────────────┘
    ↓
┌─────────────────────────────┐
│  阶段 5: 项目收尾            │ ← 关闭 Issue、输出报告
└─────────────────────────────┘
```

### 每 5 轮强制报告

系统每 5 轮迭代会自动输出进度报告，包含当前迭代次数、工具调用次数、错误次数等信息。

### 循环终止条件

| 条件 | 说明 |
|---|---|
| 所有需求完成 | 开发经理判断所有验收标准已满足 |
| 达到最大迭代 | 默认 20 轮（可在 `.env` 中配置 `MAX_ITERATIONS`） |
| 用户主动终止 | 在终端按 `Ctrl+C` |
| 无法继续 | 开发经理判断当前条件无法完成剩余需求 |

## 6. 角色配置

### 6.1 默认配置

系统使用 `config/default_roles.yaml` 作为默认角色配置。你可以复制并修改它：

```bash
cp config/default_roles.yaml my_roles.yaml
# 编辑 my_roles.yaml...
python main.py --config my_roles.yaml "你的任务"
```

### 6.2 配置文件结构

```yaml
# 全局设置
global:
  max_turns: 50                # 单次对话最大轮次
  max_iterations: 20           # 最大迭代次数
  termination_signal: "任务完成" # 终止信号
  merge_method: squash         # PR 合并方式: squash/merge/rebase

# 开发经理配置
manager:
  name: "开发经理"              # Agent 名称
  description: "..."            # 简短描述
  model: default                # 模型 (default=使用全局设置)
  system_prompt_file: config/prompts/manager_system.md  # 提示词文件
  tools:                        # 可用工具列表
    - create_issue
    - comment_on_issue
    # ...

# 开发人员配置
developer:
  # 同上...

# 测试人员配置
tester:
  # 同上...
```

### 6.3 自定义角色

你可以创建一个全新的角色配置：

```yaml
# 示例：创建一个只有 2 个角色的精简团队
global:
  max_turns: 30
  max_iterations: 10

manager:
  name: "全栈负责人"
  description: "同时负责需求管理和编码"
  tools:
    - create_issue
    - comment_on_issue
    - write_file
    - git_commit
    - git_push
    - create_pull_request
    - merge_pull_request

tester:
  name: "QA 审查员"
  description: "负责代码审查和测试"
  tools:
    - read_file
    - run_command
    - submit_pr_review
    - comment_on_issue
```

## 7. 模型后端切换

teamAgent 支持三种模型后端：

### 7.1 DeepSeek（默认）

```bash
python main.py "你的任务"
# 或显式指定:
python main.py --backend deepseek "你的任务"
```

使用 `.env` 中的 `DEEPSEEK_API_KEY` 和 `DEEPSEEK_BASE_URL`。

> ⚠️ 必须使用 `deepseek-chat`（支持 function calling），`deepseek-reasoner` 不支持工具调用。

### 7.2 OpenAI

```bash
python main.py --backend openai "你的任务"
```

使用前需要在 `.env` 中设置 OpenAI 的 API Key（可复用 `DEEPSEEK_API_KEY` 字段，或通过环境变量 `OPENAI_API_KEY` 设置）。

默认模型为 `gpt-4o`。

### 7.3 Ollama（本地）

```bash
# 先确保 Ollama 正在运行
ollama serve

# 拉取模型
ollama pull llama3

# 运行
python main.py --backend ollama "你的任务"
```

默认连接 `http://localhost:11434/v1`，使用 `llama3` 模型。

> 注意：Ollama 模型需要支持 function calling 才能正常使用工具。

## 8. 多任务并发

teamAgent 支持同时运行多个独立项目：

```bash
# 准备多个任务文件
echo "创建 Python 计算器模块" > task1.txt
echo "添加日志系统" > task2.txt
echo "编写单元测试" > task3.txt

# 批量并发执行（最大 3 个并发）
python main.py --batch task1.txt task2.txt task3.txt
```

系统使用 `asyncio.Semaphore` 控制并发数（默认最大 3 个），每个任务在独立的管道实例中运行，互不干扰。

## 10. 日志与审计

### 10.1 日志文件

日志保存在 `logs/` 目录下，分为三级：

| 文件 | 级别 | 说明 |
|---|---|---|
| `logs/team_agent_YYYY-MM-DD.log` | DEBUG+ | 详细运行日志，按天轮转，保留 7 天 |
| `logs/structured_YYYY-MM-DD.jsonl` | INFO+ | JSON 结构化日志，供日志分析 |
| `logs/errors_YYYY-MM-DD.log` | ERROR+ | 错误日志单独归档，保留 30 天 |

### 10.2 审计日志

审计日志记录所有关键操作，保存在 `logs/audit/` 目录：

```json
{
  "project_id": "24ce4e48",
  "start_time": "2026-06-12T20:00:00",
  "end_time": "2026-06-12T20:15:00",
  "total_events": 42,
  "total_tool_calls": 156,
  "total_errors": 2,
  "events": [
    {"timestamp": "...", "type": "project_start", "agent": "Pipeline", "message": "..."},
    {"timestamp": "...", "type": "phase_change", "agent": "Pipeline", "message": "CLARIFYING → DEVELOPING"}
  ],
  "tool_calls": [
    {"timestamp": "...", "agent": "开发经理", "tool": "create_issue", "args": {...}, "duration_ms": 234}
  ],
  "errors": [
    {"timestamp": "...", "agent": "Pipeline", "type": "timeout", "message": "第 5 轮迭代超时"}
  ]
}
```

### 10.3 查看日志

```bash
# 实时查看控制台日志
python main.py "任务" 2>&1 | tee run.log

# 查看最近的错误日志
cat logs/errors_$(date +%Y-%m-%d).log

# 查看审计日志
ls -la logs/audit/
cat logs/audit/audit_<project_id>_<timestamp>.json

# 使用 jq 分析结构化日志
cat logs/structured_$(date +%Y-%m-%d).jsonl | jq 'select(.record.level.name == "ERROR")'
```

## 12. 异常处理

### 12.1 自动重试

GitHub API 调用失败时，系统会自动重试（最多 3 次，指数退避）：

- GitHub 限流 (429): 等待 rate limit 重置后重试
- 服务器错误 (5xx): 指数退避重试（1s → 2s → 4s）
- 客户端错误 (4xx): **不重试**

### 12.2 Git 冲突

- 推送前自动检测与 main 的合并冲突
- 冲突时自动中止并提供解决建议
- 失败时可执行 `abort_and_reset()` 回退到安全状态

### 12.3 回退机制

- 关键操作前注册回退函数
- 失败时按注册的逆序自动执行回退
- 成功后清除回退栈

### 12.4 超时控制

| 超时类型 | 默认值 | 说明 |
|---|---|---|
| 单轮迭代 | 5 分钟 | `TimeoutGuard` |
| 需求澄清提醒 | 48 小时 | 无回复则发送提醒 |
| 需求澄清暂停 | 7 天 | 无回复则暂停项目 |
| 最大迭代 | 20 轮 | 达到上限强制终止 |

### 12.5 常见错误及解决

| 错误 | 原因 | 解决 |
|---|---|---|
| `DEEPSEEK_API_KEY 未配置` | `.env` 中未设置 API Key | 编辑 `.env` 填入密钥 |
| `GITHUB_TOKEN 未配置` | `.env` 中未设置 GitHub Token | 创建 GitHub Token 并配置 |
| `仓库不存在` | 目标仓库未被创建 | 先在 GitHub 上创建目标仓库 |
| `function calling 不支持` | 使用了不支持的模型 | 切换到 `deepseek-chat` |
| `第 N 轮迭代超时` | 单轮对话过长 | 任务可能过于复杂，拆解为更小的子任务 |

## 14. 常见问题

### Q: 如何指定开发的目标仓库？

在 `.env` 中设置 `GITHUB_REPO_OWNER` 和 `GITHUB_REPO_NAME`。例如：

```bash
GITHUB_REPO_OWNER=myusername
GITHUB_REPO_NAME=my-project
# 系统会操作 https://github.com/myusername/my-project
```

### Q: 可以不使用 GitHub 吗？

当前版本仅支持 GitHub。你可以通过修改 `src/tools/github_tools.py` 来适配其他 Git 平台。

### Q: 如何在任务进行中查看进度？

- **GitHub Issues**: 直接查看目标仓库的 Issues 页面
- **终端日志**: 控制台实时输出详细进度
- **审计日志**: 查看 `logs/audit/` 目录
- **交互模式**: 使用 `status` 命令

### Q: 支持的模型有哪些？

| 模型 | function calling | 推荐 |
|---|---|---|
| `deepseek-chat` | ✅ | 首选 |
| `deepseek-reasoner` | ❌ | 不支持 |
| `gpt-4o` | ✅ | 备选 |
| `gpt-4o-mini` | ✅ | 低成本备选 |
| `llama3` (Ollama) | ✅ | 本地运行 |

### Q: 如何自定义 Agent 的提示词？

1. 编辑 `config/prompts/` 下的 `.md` 文件
2. 或创建新的 `config/default_roles.yaml`，指定 `system_prompt_file` 指向你的提示词文件

### Q: 项目会修改本地文件吗？

会。系统在 `.workspace/` 目录下克隆和修改代码，原始项目文件不受影响。可以在 `.gitignore` 中看到 `.workspace/` 已被忽略。

### Q: 如何停止正在运行的任务？

- `Ctrl+C` 优雅退出（会执行回退和保存审计日志）
- 在 GitHub Issue 中评论 "STOP"（开发经理会检测并停止）
